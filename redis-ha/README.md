# Quick start for creating a Redis high availability cluster

The `redis-ha` Helm chart can be used to create a Redis high availability cluster with master-replica topology and HAProxy load balancer. By default this chart installs 6 pods total (3 Redis + 3 HAProxy) with Redis sentinel for automatic failover.

## About the Quick starts

The Catalog is a library of curated Helm charts to create Kubernetes resources. By default the Catalog contains a set of Helm charts provided by Application Platform for LKE (APL) to get started quickly, but they can also be modified depending on your requirements or be removed from the Catalog. The contents of the Catalog and the RBAC configuration (which Team can use which Helm chart) are managed by the platform administrator.

## Overview

This chart has been adapted for the integration into Application Platform for LKE. The following instructions have been modified to reflect the use of the Catalog.

Redis is an open source, advanced key-value store. It is often referred to as a data structure server since keys can contain strings, hashes, lists, sets and sorted sets.

[Overview of Redis](http://redis.io)

## How to use this quick start

1. Go to the `values` tab and fill in a name for your Redis HA cluster
2. Optional, but recommended: Create a (sealed) secret that you share between the Redis cluster and the workloads using it. Add this to `existingSecret` (secret name) and `authKey` (secret key). This prevents that the autogenerated secret from Redis gets cycled, while the application runs with a stale credential.
3. Optional: Change the `replicas` count. Default is set to `3`.
4. Optional: Disable HAProxy load balancer by setting `haproxy.enabled: false` (enabled by default)
5. Optional: Configure Prometheus monitoring (see Monitoring Integration section for details)

## Using the secret

Use a `secretKeyRef` to configure the environment variables for the container to use the Redis cluster. The autogenerated secret is used as follows. It is however encouraged to create and use your own sealed secret.

```yaml
- name: REDIS_PASSWORD
  valueFrom:
    secretKeyRef:
      name: <cluster-name>  # Replace with your cluster name
      key: auth
```

## Configuration Details

### High Availability Architecture

This chart deploys a Redis high availability setup with:

- **Redis Master-Replica Topology**: Multiple Redis instances with automatic master election
- **Redis Sentinel**: Monitors Redis instances and handles failover
- **HAProxy Load Balancer**: Provides connection load balancing and read/write splitting (enabled by default)
- **Persistent Storage**: Data persistence across pod restarts
- **Split-brain Detection**: Automatic detection and recovery from split-brain scenarios

### Resource Management

Setting resource requests and limits is essential for production workloads. Configure these based on your specific use case:

```yaml
redis:
  resources:
    limits:
      cpu: 200m
      memory: 256Mi
    requests:
      cpu: 100m
      memory: 128Mi
```

### Redis Version Configuration

To use a different Redis version, specify it using the `image.tag` parameter:

```yaml
image:
  repository: public.ecr.aws/docker/library/redis
  tag: "7.2.4-alpine"
```

### Security and TLS

Note that Application Platform for LKE provides mutual TLS through the Istio service mesh, so TLS is not required for transport security alone.

TLS support can be enabled for Redis itself by configuring the `tls.*` parameters if needed for specific security requirements.

### Monitoring Integration

Enable Prometheus metrics collection and Platform Prometheus integration:

```yaml
exporter:
  enabled: true
  serviceMonitor:
    enabled: true
    labels:
      prometheus: system
```

### Performance Tuning

For high-performance deployments, Redis may require kernel-level optimizations:

```yaml
sysctlImage:
  enabled: true
  mountHostSys: true
  command:
    - /bin/sh
    - -c
    - |-
      sysctl -w net.core.somaxconn=10000
      echo never > /host-sys/kernel/mm/transparent_hugepage/enabled
```

## Troubleshooting

### Common Configuration Issues

1. **Authentication Setup**: When using `existingSecret`, ensure the secret contains the key specified in `authKey` (default: "auth")

2. **Resource Constraints**: Monitor memory usage as Redis keeps all data in memory. Set appropriate resource limits based on your dataset size

3. **Sentinel Quorum**: The default quorum of 2 works well for 3 replicas. Adjust `sentinel.quorum` if you change the replica count

4. **Persistent Storage**: Ensure your cluster has dynamic volume provisioning or manually create persistent volumes

5. **Network Connectivity**: When using network policies, verify that applications can reach the Redis service on the configured ports (6379 for Redis, 26379 for Sentinel)

### High Availability Behavior

- **Automatic Failover**: Redis Sentinel automatically promotes a replica to master if the current master fails
- **Split-brain Prevention**: The chart includes detection and recovery mechanisms for split-brain scenarios
- **Load Balancing**: When HAProxy is enabled, it provides intelligent routing to the current master for writes and load balancing for reads
